# Pusher Integration

dotdo provides two ways to work with Pusher-compatible real-time APIs:

| | @dotdo/pusher | pusher.do |
|---|----------------|----------|
| **Type** | OSS npm package | Managed edge service |
| **Backend** | In-memory channels | Durable Object SQLite |
| **Persistence** | None (resets on restart) | Full durability |
| **Use case** | Testing, development | Production workloads |
| **Infrastructure** | Self-hosted Worker | Zero (we manage it) |
| **Install** | `npm install @dotdo/pusher` | Deploy to Cloudflare |

## Quick Comparison

```typescript
// @dotdo/pusher - Self-hosted, great for testing
import Pusher from '@dotdo/pusher'

const pusher = new Pusher('app-key', {
  cluster: 'mt1',
  wsHost: 'your-worker.workers.dev',
})

const channel = pusher.subscribe('my-channel')
channel.bind('event', (data) => console.log(data))
```

```typescript
// pusher.do - Managed edge service
import Pusher from 'pusher.do'

const pusher = new Pusher('your-app-key', {
  cluster: 'global', // Edge routing
})

const channel = pusher.subscribe('my-channel')
channel.bind('event', (data) => console.log(data))
```

## When to Use Each

### Use @dotdo/pusher when:

- **Unit testing** - Fast, isolated tests without external dependencies
- **Development** - Quick iteration with local Wrangler dev
- **Self-hosting** - Need full control over infrastructure
- **Cost optimization** - Running on existing Cloudflare Workers

<Callout type="info">
@dotdo/pusher is a drop-in replacement for pusher-js. Same API, self-hosted on your own Cloudflare Workers with Durable Objects.
</Callout>

### Use pusher.do when:

- **Production applications** - Managed infrastructure, zero ops
- **Global scale** - Edge deployment in 300+ cities
- **Unlimited connections** - No per-connection fees
- **Enterprise features** - Webhooks, analytics, monitoring

<Callout type="info">
pusher.do is Pusher on the Edge - a fully managed service with unlimited connections and no per-message fees.
</Callout>

## Migration Path

Start with `@dotdo/pusher` for development, then switch to `pusher.do` for production:

```typescript
// Development (self-hosted)
import Pusher from '@dotdo/pusher'
const pusher = new Pusher('app-key', {
  wsHost: 'localhost:8787',
})

// Production (managed service) - same API!
import Pusher from 'pusher.do'
const pusher = new Pusher('your-app-key', {
  cluster: 'global',
})

// Your application code stays the same
const channel = pusher.subscribe('notifications')
channel.bind('new-message', (data) => {
  showNotification(data.title, data.body)
})
```

Both packages share the same Pusher-compatible API, making migration seamless.
