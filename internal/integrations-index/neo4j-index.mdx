# Neo4j Integration

dotdo provides two ways to work with Neo4j-compatible graph database APIs:

| | @dotdo/neo4j | neo4j.do |
|---|--------------|----------|
| **Type** | OSS npm package | Managed edge service |
| **Storage** | In-memory | Durable Object SQLite |
| **Persistence** | None (resets on restart) | Full durability |
| **Use case** | Testing, prototyping | Production workloads |
| **Infrastructure** | Zero | Zero (we manage it) |
| **Install** | `npm install @dotdo/neo4j` | Deploy to Cloudflare |

## Quick Comparison

```typescript
// @dotdo/neo4j - In-memory, great for tests
import neo4j from '@dotdo/neo4j'

const driver = neo4j.driver('bolt://localhost', neo4j.auth.basic('neo4j', 'password'))
const session = driver.session()

await session.run('CREATE (n:Person {name: $name})', { name: 'Alice' })
// Data lost on restart
```

```typescript
// neo4j.do - Persistent edge graph database
import { GraphClient } from 'neo4j.do'

const client = new GraphClient('https://your-worker.workers.dev')
const session = client.session()

await session.run('CREATE (n:Person {name: $name})', { name: 'Alice' })
// Data persists globally
```

## When to Use Each

### Use @dotdo/neo4j when:

- **Unit testing** - Fast, isolated tests without external dependencies
- **Prototyping** - Quick iteration without infrastructure setup
- **Edge functions** - Need Neo4j API but no persistence required
- **CI/CD pipelines** - Tests that need Cypher and graph semantics

<Callout type="info">
@dotdo/neo4j is a complete in-memory Neo4j implementation with full Cypher support. Perfect for tests and development, but data doesn't persist between restarts.
</Callout>

### Use neo4j.do when:

- **Production applications** - Real graph data that must persist
- **Global edge deployment** - Data close to users worldwide
- **AI/Agent workloads** - Graph-based reasoning, knowledge graphs
- **Real-time features** - Live graph traversals and recommendations

<Callout type="info">
neo4j.do is Neo4j on the Edge - a fully managed graph database running on Cloudflare's global network with zero infrastructure to manage.
</Callout>

## Migration Path

Start with `@dotdo/neo4j` for development, then upgrade to `neo4j.do` for production:

```typescript
// Development (package)
import neo4j from '@dotdo/neo4j'
const driver = neo4j.driver('bolt://localhost', neo4j.auth.basic('neo4j', 'password'))
const session = driver.session()

// Production (service) - same API!
import { GraphClient } from 'neo4j.do'
const client = new GraphClient('https://your-neo4j.workers.dev')
const session = client.session()

// Your application code stays the same
const result = await session.run('MATCH (n:Person)-[:KNOWS]->(friend) RETURN n, friend')
const records = await result.records()
```

Both packages share the same Neo4j-compatible API, making migration seamless.

## Architecture Comparison

### @dotdo/neo4j

```
+------------------------------------------+
|            Your Application              |
|                                          |
|  import neo4j from '@dotdo/neo4j'        |
+------------------------------------------+
                    |
                    v
+------------------------------------------+
|           In-Memory Graph Store          |
|   (Cypher Parser + Graph Algorithms)     |
+------------------------------------------+
```

### neo4j.do

```
+------------------------------------------+
|            Your Application              |
|                                          |
|  import { GraphClient } from 'neo4j.do'  |
+------------------------------------------+
                    |
                    v
+------------------------------------------+
|           neo4j.do Worker (Edge)         |
+------------------------------------------+
|  Query Translator  |  Graph Algorithms   |
+------------------------------------------+
|      Durable Object SQLite Storage       |
+------------------------------------------+
|  Vectorize (Embeddings)  |  R2 (Backups) |
+------------------------------------------+
```

Both approaches support the same graph patterns (social networks, knowledge graphs, recommendations) - choose based on whether you need persistence and production features.
