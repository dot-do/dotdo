# GitHub Integration

dotdo provides two ways to work with GitHub APIs:

| | @dotdo/github | github.do |
|---|---------------|-----------|
| **Type** | OSS npm package | Managed edge service |
| **API Style** | Octokit-compatible | Native edge API |
| **Runtime** | Edge-compatible | Cloudflare Workers |
| **Use case** | Testing, migration | Production workloads |
| **Infrastructure** | Zero | Zero (we manage it) |
| **Install** | `npm install @dotdo/github` | Deploy to Cloudflare |

## Quick Comparison

```typescript
// @dotdo/github - Octokit-compatible, great for migration
import { Octokit } from '@dotdo/github'

const octokit = new Octokit({ auth: env.GITHUB_TOKEN })

const { data: repo } = await octokit.rest.repos.get({
  owner: 'octocat',
  repo: 'hello-world',
})
```

```typescript
// github.do - Native edge GitHub integration
import { GitHub } from 'github.do'

const github = new GitHub(env)

const repo = await github.repo('octocat/hello-world')
const issues = await github.issues.list({ state: 'open' })
```

## When to Use Each

### Use @dotdo/github when:

- **Migrating from Octokit** - Drop-in replacement with same API
- **Unit testing** - Mock GitHub API in tests without network calls
- **Gradual migration** - Keep using Octokit patterns while moving to edge
- **Existing codebases** - Minimal changes to existing GitHub integrations

<Callout type="info">
@dotdo/github is an Octokit-compatible library. Use it when you need the familiar Octokit API or are migrating existing code.
</Callout>

### Use github.do when:

- **Edge-native workflows** - Built for Cloudflare Workers from the ground up
- **Webhook handling** - Built-in signature verification and typed handlers
- **GitHub App hosting** - Managed app installation and authentication
- **CI/CD automation** - Durable workflow triggers and status checks
- **Real-time sync** - Bidirectional sync with GitHub repositories

<Callout type="info">
github.do is GitHub on the Edge - a fully managed service running on Cloudflare's global network with built-in webhook handling, App authentication, and workflow automation.
</Callout>

## Migration Path

Start with `@dotdo/github` for drop-in Octokit compatibility, then migrate to `github.do` for managed features:

```typescript
// Phase 1: Use @dotdo/github (drop-in for Octokit)
import { Octokit } from '@dotdo/github'

const octokit = new Octokit({ auth: env.GITHUB_TOKEN })

const { data: issue } = await octokit.rest.issues.create({
  owner: 'my-org',
  repo: 'my-repo',
  title: 'Bug report',
  body: 'Something is broken',
})
```

```typescript
// Phase 2: Migrate to github.do for managed features
import { GitHub } from 'github.do'

const github = new GitHub(env)

const issue = await github.issues.create('my-org/my-repo', {
  title: 'Bug report',
  body: 'Something is broken',
})

// Plus: automatic rate limiting, caching, and webhook routing
```

## Common Use Cases

### Webhook Automation

Both packages support webhook handling, but with different approaches:

```typescript
// @dotdo/github - Manual webhook handler
import { createWebhookHandler } from '@dotdo/github'

const handler = createWebhookHandler({
  push: async (payload) => {
    console.log('Push to:', payload.ref)
  },
  pull_request: async (payload) => {
    if (payload.action === 'opened') {
      // Add reviewers, labels, etc.
    }
  },
}, { secret: env.GITHUB_WEBHOOK_SECRET })

export default { fetch: handler }
```

```typescript
// github.do - Declarative webhook routing
import { GitHubApp } from 'github.do'

export default GitHubApp({
  on: {
    'push': async (ctx) => {
      // Automatic signature verification
      console.log('Push to:', ctx.payload.ref)
    },
    'pull_request.opened': async (ctx) => {
      // Scoped event handlers
      await ctx.github.issues.addLabels(['needs-review'])
    },
  },
})
```

### CI/CD Integration

```typescript
// Trigger deployment on merge to main
import { Octokit, createWebhookHandler } from '@dotdo/github'

const octokit = new Octokit({ auth: env.GITHUB_TOKEN })

const handler = createWebhookHandler({
  push: async (payload) => {
    if (payload.ref !== 'refs/heads/main') return

    await octokit.rest.actions.createWorkflowDispatch({
      owner: payload.repository.owner.login,
      repo: payload.repository.name,
      workflow_id: 'deploy.yml',
      ref: 'main',
    })
  },
}, { secret: env.GITHUB_WEBHOOK_SECRET })
```

Both packages share the same webhook event types and payload structures, making migration straightforward.
