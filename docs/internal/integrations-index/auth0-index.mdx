# Auth0 Integration

dotdo provides two ways to work with Auth0-compatible authentication APIs:

| | @dotdo/auth0 | auth0.do |
|---|--------------|----------|
| **Type** | OSS npm package | Managed edge service |
| **Storage** | In-memory or configurable | Durable Object SQLite |
| **Persistence** | Configurable | Full durability |
| **Use case** | Testing, development | Production workloads |
| **Infrastructure** | Self-managed | Zero (we manage it) |
| **Install** | `npm install @dotdo/auth0` | Deploy to Cloudflare |

## Quick Comparison

```typescript
// @dotdo/auth0 - Drop-in SDK replacement for testing
import { ManagementClient, AuthenticationClient } from '@dotdo/auth0'

const management = new ManagementClient({
  domain: 'your-tenant.auth0.com',
  clientId: 'your-client-id',
  clientSecret: 'your-client-secret',
})

const user = await management.users.create({
  email: 'alice@example.com',
  password: 'SecureP@ssw0rd!',
  connection: 'Username-Password-Authentication',
})
```

```typescript
// auth0.do - Managed edge authentication
import { ManagementClient } from 'auth0.do'

const management = new ManagementClient({
  domain: 'your-app.auth0.do',
  clientId: 'your-client-id',
  clientSecret: 'your-client-secret',
})

// Same API, globally distributed storage
const user = await management.users.create({
  email: 'alice@example.com',
  password: 'SecureP@ssw0rd!',
  connection: 'Username-Password-Authentication',
})
```

## When to Use Each

### Use @dotdo/auth0 when:

- **Unit testing** - Fast, isolated tests without external dependencies
- **Local development** - Develop auth flows without Auth0 credentials
- **CI/CD pipelines** - Tests that need Auth0 SDK semantics
- **Migration testing** - Validate code before switching from Auth0

<Callout type="info">
@dotdo/auth0 is a complete Auth0 SDK reimplementation. Perfect for tests and development where you need Auth0-compatible APIs without external services.
</Callout>

### Use auth0.do when:

- **Production applications** - Real user data that must persist globally
- **Global edge deployment** - Authentication close to users worldwide
- **Cost optimization** - Escape Auth0's per-MAU pricing
- **Data sovereignty** - Self-hosted auth with full data control

<Callout type="info">
auth0.do is Auth0 on the Edge - a fully managed authentication service running on Cloudflare's global network with zero infrastructure to manage.
</Callout>

## Migration Path

Start with `@dotdo/auth0` for development, then upgrade to `auth0.do` for production:

```typescript
// Development (package)
import { ManagementClient } from '@dotdo/auth0'

const management = new ManagementClient({
  domain: 'dev.auth0.do',
  clientId: 'dev-client-id',
  clientSecret: 'dev-secret',
})

// Production (service) - same API!
import { ManagementClient } from 'auth0.do'

const management = new ManagementClient({
  domain: 'your-app.auth0.do',
  clientId: 'prod-client-id',
  clientSecret: 'prod-secret',
})

// Your application code stays the same
const user = await management.users.get({ id: 'user_123' })
```

Both packages share the same Auth0-compatible API, making migration seamless.

## Comparison with Auth0 Cloud

| Auth0 Cloud | auth0.do |
|-------------|----------|
| Per-MAU pricing ($23/1K MAU) | Flat resource pricing |
| Single-region data residency | Edge-local in 300+ cities |
| Vendor lock-in | Self-hosted, portable |
| Rate-limited APIs | Unlimited API calls |
| Separate auth infrastructure | Unified with your DO data |

**This is a reimplementation.** Your auth data lives in Durable Object SQLite - not proxied to Auth0 servers. The domain and credentials are used for namespacing and JWT configuration.
