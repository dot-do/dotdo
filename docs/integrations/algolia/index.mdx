# Algolia Integration

dotdo provides two ways to work with Algolia-compatible search APIs:

| | @dotdo/algolia | algolia.do |
|---|----------------|------------|
| **Type** | OSS npm package | Managed edge service |
| **Storage** | In-memory | Durable Object SQLite |
| **Persistence** | None (resets on restart) | Full durability |
| **Use case** | Testing, prototyping | Production workloads |
| **Infrastructure** | Zero | Zero (we manage it) |
| **Install** | `npm install @dotdo/algolia` | Deploy to Cloudflare |

## Quick Comparison

```typescript
// @dotdo/algolia - In-memory, great for tests
import algoliasearch from '@dotdo/algolia'

const client = algoliasearch('my-app', 'my-key')
const index = client.initIndex('products')
await index.saveObject({ objectID: '1', name: 'Widget' })
// Data lost on restart
```

```typescript
// algolia.do - Persistent edge search
import algoliasearch from 'algolia.do'

const client = algoliasearch('https://your-worker.workers.dev')
const index = client.initIndex('products')
await index.saveObject({ objectID: '1', name: 'Widget' })
// Data persists globally
```

## When to Use Each

### Use @dotdo/algolia when:

- **Unit testing** - Fast, isolated tests without external dependencies
- **Prototyping** - Quick iteration without infrastructure setup
- **Edge functions** - Need Algolia API but no persistence required
- **CI/CD pipelines** - Tests that need Algolia semantics

<Callout type="info">
@dotdo/algolia is a complete in-memory Algolia implementation. Perfect for tests and development, but data doesn't persist between restarts.
</Callout>

### Use algolia.do when:

- **Production applications** - Real user data that must persist
- **Global edge deployment** - Search index close to users worldwide
- **Full-text search** - FTS5-powered search with scoring and highlights
- **Drop-in replacement** - Works with InstantSearch and other Algolia frontend libraries

<Callout type="info">
algolia.do is Algolia on the Edge - a fully managed service running on Cloudflare's global network with zero infrastructure to manage.
</Callout>

## Feature Comparison

| Feature | @dotdo/algolia | algolia.do |
|---------|----------------|------------|
| Search | Full-text with prefix | Full-text with FTS5 |
| Filters | AND/OR/NOT, numeric, facet | AND/OR/NOT, numeric, facet |
| Facets | Count aggregation | Count aggregation |
| Highlighting | Pre/post tags, match levels | Pre/post tags, match levels |
| Multi-index | multipleQueries, multipleGetObjects | multipleQueries, multipleGetObjects |
| Settings | searchableAttributes, faceting | searchableAttributes, faceting |
| InstantSearch | Compatible | Compatible |
| Synonyms | Not yet | Planned |
| Query Rules | Not yet | Planned |
| Geo search | Types only | Planned |

## Migration Path

Start with `@dotdo/algolia` for development, then upgrade to `algolia.do` for production:

```typescript
// Development (package)
import algoliasearch from '@dotdo/algolia'
const client = algoliasearch('my-app', 'my-key')
const index = client.initIndex('products')

// Production (service) - same API!
import algoliasearch from 'algolia.do'
const client = algoliasearch('https://your-algolia.workers.dev')
const index = client.initIndex('products')

// Your application code stays the same
const results = await index.search('wireless headphones')
```

Both packages share the same Algolia-compatible API, making migration seamless.
