# Installation

Two minutes. That's all you need.

## Prerequisites

Before you begin, make sure you have:

| Requirement | Version | Check Command |
|-------------|---------|---------------|
| Node.js | 18.0.0+ | `node --version` |
| npm/pnpm/yarn/bun | Latest | `npm --version` |
| Cloudflare account | Free tier works | [Sign up](https://dash.cloudflare.com/sign-up) |

<Callout type="info">
**Bun users:** dotdo works great with Bun 1.1.0+. The CLI uses Bun internally for faster execution.
</Callout>

## Create a New Project

The fastest way to get started is with `npx`:

<Tabs items={['npm/npx', 'pnpm', 'yarn', 'bun']}>
  <Tab value="npm/npx">
```bash
npx dotdo@latest init my-startup
cd my-startup
npm install
```
  </Tab>
  <Tab value="pnpm">
```bash
pnpm dlx dotdo@latest init my-startup
cd my-startup
pnpm install
```
  </Tab>
  <Tab value="yarn">
```bash
yarn dlx dotdo@latest init my-startup
cd my-startup
yarn install
```
  </Tab>
  <Tab value="bun">
```bash
bunx dotdo@latest init my-startup
cd my-startup
bun install
```
  </Tab>
</Tabs>

<Callout type="info">
**How it works:** `npx dotdo@latest init` downloads and runs the dotdo CLI temporarily. You don't need to install dotdo globally - the CLI is included as a project dependency.
</Callout>

### Expected Output

When the init command succeeds, you'll see:

```
Project "my-startup" created successfully!

Next steps:
  cd my-startup
  npm install
  npm run dev
```

### Initialize in Current Directory

To initialize dotdo in an existing directory:

```bash
mkdir my-startup && cd my-startup
npx dotdo@latest init .
npm install
```

## What Got Created?

After running `init`, your project contains:

```
my-startup/
  src/
    index.ts        # Your Durable Object class and worker entry point
  wrangler.jsonc    # Cloudflare Workers configuration
  package.json      # Project dependencies and scripts
  tsconfig.json     # TypeScript configuration
```

Here's what each file does:

| File | Purpose |
|------|---------|
| `src/index.ts` | Your main Durable Object class. This is where your business logic lives. Extends `DO` (basic) or `Startup` (with AI agents). |
| `wrangler.jsonc` | Cloudflare Workers configuration. Defines your DO bindings, SQLite migrations, and deployment settings. |
| `package.json` | Project metadata and dependencies. Includes `dotdo` as a dependency and scripts for dev/deploy. |
| `tsconfig.json` | TypeScript settings optimized for Cloudflare Workers (ES2022 target, bundler resolution). |

### The Generated Code

The init command creates a minimal Durable Object:

```typescript title="src/index.ts"
import { DO } from 'dotdo'

export class MyStartup extends DO {
  static readonly $type = 'MyStartup'
}

// Re-export the simple worker as default
export { default } from 'dotdo/workers/simple'
```

This gives you:
- SQLite persistence (automatic)
- REST API routing (automatic)
- State management (automatic)

## Project Configuration

The init command creates `wrangler.jsonc` with sensible defaults:

```json title="wrangler.jsonc"
{
  "name": "my-startup",
  "main": "src/index.ts",
  "compatibility_date": "2024-12-01",
  "compatibility_flags": ["nodejs_compat"],
  "durable_objects": {
    "bindings": [
      { "name": "DO", "class_name": "MyStartup" }
    ]
  },
  "migrations": [
    { "tag": "v1", "new_sqlite_classes": ["MyStartup"] }
  ]
}
```

Key settings explained:

| Setting | Value | Why |
|---------|-------|-----|
| `compatibility_flags` | `["nodejs_compat"]` | Enables Node.js APIs in Workers |
| `new_sqlite_classes` | Your DO class | Enables built-in SQLite for persistence |

<Callout type="warn">
**First time using Wrangler?** Run `npx wrangler login` to authenticate with your Cloudflare account.
</Callout>

## Verify Installation

Start the development server:

```bash
npm run dev
```

You should see output similar to:

```
 ⛅️ wrangler 3.99.0
-------------------

Starting local server...
[wrangler:inf] Ready on http://localhost:8787
```

Test your API:

```bash
curl http://localhost:8787
```

You should receive a JSON response from your Durable Object.

## Environment Variables

Create a `.dev.vars` file for local secrets:

```bash title=".dev.vars"
OPENAI_API_KEY=sk-...
ANTHROPIC_API_KEY=sk-ant-...
```

<Callout type="warn">
**These keys are required for AI agents.** Without at least one API key, the named agents (Priya, Ralph, Tom, etc.) can't call LLMs. See [API Keys](/docs/getting-started/api-keys) for details.
</Callout>

<Callout type="error">
Never commit `.dev.vars`. Add it to `.gitignore` if not already present.
</Callout>

For production, use Wrangler secrets:

```bash
npx wrangler secret put OPENAI_API_KEY
npx wrangler secret put ANTHROPIC_API_KEY
```

## TypeScript Configuration

The init command creates the right `tsconfig.json`:

```json title="tsconfig.json"
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "types": ["@cloudflare/workers-types"]
  }
}
```

Key settings:

| Option | Value | Why |
|--------|-------|-----|
| `target` | ES2022 | V8 isolates support modern JS |
| `moduleResolution` | bundler | Wrangler bundles your code |
| `types` | workers-types | Cloudflare globals like `DurableObject` |

## Upgrade to Startup Class

The init command creates a basic `DO` class. To build an AI-powered startup with agents, upgrade to the `Startup` class:

```typescript title="src/index.ts"
import { Startup } from 'dotdo'
import { priya, ralph, tom, mark, sally } from 'agents.do'

export class MyStartup extends Startup {
  hypothesis = 'Your hypothesis here'

  async launch() {
    const spec = priya`define the MVP for ${this.hypothesis}`
    let app = ralph`build ${spec}`

    do {
      app = ralph`improve ${app} per ${tom}`
    } while (!await tom.approve(app))

    mark`announce the launch`
    sally`start selling`
  }
}

export { default } from 'dotdo/workers/simple'
```

<Callout type="info">
**DO vs Startup:** `DO` is the base Durable Object class providing REST routing, SQLite persistence, and state management. `Startup` extends `DO` with AI agent integration, human escalation, and workflow capabilities. Use `DO` for simple services; use `Startup` when you need AI agents.
</Callout>

## Import Aliases

The dotdo ecosystem uses domain-style imports like `agents.do` and `humans.do`. These are **path aliases** configured automatically by `npx dotdo init`.

<Callout type="warn">
**Required for named agents and human escalation.** Without these aliases, you'll get "Cannot find module 'agents.do'" errors.
</Callout>

### Manual Setup (Existing Projects)

If you added dotdo to an existing project, configure the aliases manually.

Add path mappings to your `tsconfig.json`:

```json title="tsconfig.json"
{
  "compilerOptions": {
    "paths": {
      "agents.do": ["./node_modules/dotdo/dist/agents/named/index.js"],
      "humans.do": ["./node_modules/dotdo/dist/lib/humans/index.js"]
    }
  }
}
```

Or use Node.js subpath imports in `package.json`:

```json title="package.json"
{
  "imports": {
    "agents.do": "dotdo/agents/named",
    "humans.do": "dotdo/lib/humans"
  }
}
```

Now you can use the domain-style imports:

```typescript
import { priya, ralph, tom, mark, sally, quinn } from 'agents.do'
import { ceo, legal, cfo, cto, hr, support } from 'humans.do'
```

|---------|-------------|
| `npm run dev` | Start local development server |

## Troubleshooting

### "Cannot find module 'agents.do'" or "Cannot find module 'humans.do'"

These are path aliases, not npm packages. Add the path mappings to your `tsconfig.json` or `package.json` as shown in the [Import Aliases](#import-aliases) section.

### "Cannot find module 'dotdo'"

Make sure you're in the project directory and ran `npm install`.

### "Wrangler not authenticated"

Run `npx wrangler login` and follow the browser prompts.

### "Durable Object not found"

Check that your `wrangler.jsonc` has the correct class name in the `durable_objects` binding.

### Port 8787 already in use

Another Wrangler instance is running. Kill it or use a different port:

```bash
npm run dev -- --port 8788
```

### "Could not find bun"

The dotdo CLI requires Bun. Install it as a dev dependency or globally:

```bash
npm install -D bun
# or
curl -fsSL https://bun.sh/install | bash
```
